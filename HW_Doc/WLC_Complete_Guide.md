# WLC 项目完整教程

这是一份专门写给“小白接手项目”场景的统一教程，用来替代之前分散的多份 HTML 教程。  
你不需要先懂 Qi 协议，也不需要先熟悉这个仓库。只要你准备开始接手这个项目，这份教程的目标就是把下面这些事情按顺序讲清楚：

- 这个项目到底是做什么的
- 它跑在什么硬件上
- 工程目录该怎么读
- 程序是怎么启动起来的
- 协议流程是怎么往前走的
- 出问题时应该先看哪里

你可以把它理解成一份“从零开始接手项目”的带路教程，而不是只给熟手看的速查表。

---

## 1. 项目是什么

先用一句最直白的话说：

这个项目是一个“无线充电发射端”的单片机固件工程。  
它的任务，是让一块带线圈的硬件板子，按照 Qi 协议给手机这类接收端设备供电。

如果你之前没接触过“发射端”和“接收端”这两个说法，可以先这样记：

- 发射端：你手上的这块板子，它负责发能量
- 接收端：手机或别的被充电设备，它负责收能量

这个项目真正要做的事情，可以拆成 3 步：

1. 让 MCU 控制功率桥和线圈，把能量发出去。
2. 接收手机回传的通信数据，知道对方现在是什么状态。
3. 按 Qi 协议一步一步推进流程，直到开始稳定充电，或者在异常时结束传输。

如果还想再压缩成一句更方便记忆的话，可以记成：

`这个工程 = 无线充发射端硬件控制 + Qi 协议流程 + FreeRTOS 周期调度`

## 2. 先建立整体认知

如果你是第一次接这个工程，不要急着点开几十个 `.c` 文件。  
先把下面 4 个“总认知”记住，后面读代码会轻松很多。

### 2.0 先统一几个最常出现的叫法

为了避免你读到后面时被不同说法绕晕，这里先把全文最常出现的几个词统一一下：

- “发射端”：指你现在维护的这块无线充电板，也就是本项目本身
- “接收端”：指手机或别的被充电设备
- “上层”：默认指更靠近协议逻辑的一侧，不是硬件底层
- “抽象接口层”：默认指 `User/WLC_driver/` 这一层
- “服务层”：默认指 `User/WLC_server/` 这一层
- “应用层”或“主状态机”：默认指 `User/WLC_app/` 里的协议阶段逻辑
- “协议栈”：在这份教程里，通常指 `wlc_lib` 加上应用层阶段逻辑一起工作的整体
- “校准”：如果没有特别说明，默认是指 `EPP` 进入正式功率传输前的那段校准子流程

后面如果你看到这些词反复出现，就按这里的含义来理解。

### 2.1 它不是网页项目

仓库里以前那些 HTML 文件，都是教程，不是项目运行界面。  
这个项目真正跑起来的时候，是运行在单片机里的，不是在浏览器里跑页面。

### 2.2 它已经是“移植完成态”

你现在看到的，不是一套“等你自己从零移植”的空模板。  
它已经是一套整理过、能继续维护的工程。

所以你的阅读重点应该放在：

- 当前工程怎么组织
- 当前工程怎么启动
- 当前工程每一层在做什么

而不是一直纠结“它历史上是怎么移植过来的”。

### 2.3 它是明显分层的

这个工程不是“所有代码乱堆在一个目录里”的写法，而是有明显分层的。

- `Bsp/`：直接碰硬件
- `User/WLC_driver/`：做硬件接口抽象
- `User/WLC_server/`：做协议无关基础服务
- `User/WLC_app/`：做 Qi 协议状态机

### 2.4 它不是裸机 while(1)，而是 FreeRTOS 工程

主循环不是直接在 `main()` 里写一个死循环完成的，而是放在 FreeRTOS 任务里周期运行的。

如果你不知道该从哪个目录开始看，这一章很重要。  
因为对新手来说，先搞懂“文件大概放在哪一层”，比先记住每个函数干了什么更有用。

下面这个目录结构，是阅读当前项目时最应该先建立起来的脑图：

```text
wlc_code/
├── Bsp/                         板级驱动层
├── Components/ringbuffer/       串口环形缓冲组件
├── Core/                        入口与任务创建
├── Drivers/                     芯片外设库与 CMSIS

├── HW_Doc/                      教程与硬件资料
├── Lib/                         Tomato_wlc 静态库与头文件
├── MDK-ARM/                     Keil 工程文件与构建产物
└── User/

    ├── WLC_driver/              硬件抽象接口层
    ├── WLC_server/              软件定时器、ASK 解调、功率桥
    └── WLC_app/                 Qi 协议应用层
```

### 5.1 `Core/`

- `Core/Src/main.c`
- `Core/Src/freertos_task.c`

这里是程序入口层。  
如果你想知道“上电之后程序先做了什么”，通常先从这里看。

### 5.2 `Bsp/`

这里放的是和当前硬件板子绑定最紧的代码。  
如果以后你换板子、改引脚、改外设，最容易动到的往往就是这里。

- UART
- Timer
- IRQ
- PWM
- ADC
- QC
- EEPROM

### 5.3 `User/WLC_driver/`

这里可以理解成“协议逻辑”和“底层硬件”之间的一层隔离带。  
上层代码尽量不直接碰寄存器，而是走这里封装好的抽象接口。

### 5.4 `User/WLC_server/`

这里放的是一些协议运行时会反复用到、但又不属于某个单独阶段的公共服务模块：

- `wlc_timer`
- `wlc_physical`
- `wlc_power_bridge`

### 5.5 `User/WLC_app/`

这里是最核心的协议应用层。  
如果你后面开始读协议流程，大部分关键阶段代码都会在这里。

它负责：

- 维护阶段状态机
- 处理每个阶段收到的包
- 决定何时切换到下一阶段

### 5.6 用“哪些代码尽量别动”来理解目录

除了知道“代码放在哪”，还要知道“哪些地方不要一上来就乱改”。  
对新手来说，这一点非常重要。

- 基本不动：`Drivers/`、`FreeRTOS/`、`Lib/`、协议核心源码
- 按硬件板子适配：`Bsp/`
- 负责把上下层接起来：启动入口、接口绑定、回调注册

你可以先用下面这个粗粒度规则做第一轮定位：

- 协议阶段异常，优先看 `User/WLC_app/`
- 解调和功率桥异常，优先看 `User/WLC_server/`
- 引脚、PWM、ADC、IRQ 异常，优先看 `Bsp/`

## 6. 项目入口和启动流程

以下内容根据源码结构整理补充

如果你以前习惯从 `main()` 往下追，这一章会帮你少走很多弯路。  
因为这个工程虽然也有 `main()`，但真正重要的初始化并不都写在里面。

当前工程的实际入口可以先概括成“表面简单，真正逻辑在任务里展开”：

### 6.1 `main.c`

`main()` 现在只做两件事：

1. `App_Task_Create()`
2. `vTaskStartScheduler()`

这说明：真正的初始化细节，不是在 `main.c` 里一行行铺开的，而是在后面的任务函数里完成的。

### 6.2 `freertos_task.c`

这个文件才是系统真正启动后的关键展开点。  
如果你想知道“上电后程序到底一步一步干了什么”，就要重点看这里。

`app_1ms_task()` 在启动时依次做这些事情：

1. 初始化 `bsp_timer`
2. 初始化 `bsp_uart`
3. 初始化 `bsp_irq`
4. 初始化 `bsp_pwm`
5. 初始化 `bsp_adc`
6. 初始化 `bsp_Qc`
7. 进行 `QC 9V / 12V` 相关诱骗设置
8. 注册 timer 和 ASK 中断回调
9. 通过 `wlc_lib_init()` 初始化 WLC
10. 每 1ms 调用一次 `wlc_lib_handler()`

以下内容根据源码结构整理补充

这里要特别注意一件事：  
当前启动代码里，并没有直接写一行“现在去跑应用层主状态机”。  
真正的流程，是通过 `wlc_lib_init()` 和后续周期调用 `wlc_lib_handler()` 逐步带起来的。

也就是说，教程里虽然可以用“最终会跑到应用层状态机”来帮助理解，但如果你在源码里找“app_1ms_task 直接调用 `wlc_app_task()`”这一行，是找不到的。

可以把整个流程记成：

```text
main
-> 创建任务
-> 启动调度器
-> app_1ms_task 初始化外设
-> 注入 wlc_lib_api_t
-> wlc_lib_init
-> 由 wlc_lib 内部带起应用层初始化
-> 后续通过 wlc_lib_handler 周期推进协议栈
```

## 7. FreeRTOS 在这个工程里怎么工作

如果你以前只写过“`main()` 里一个大循环跑到底”的单片机程序，那这一章要先看明白。  
因为这个项目不是那种写法，它把主要工作分散在 FreeRTOS 任务里按节拍运行。

这一章不打算把 FreeRTOS 理论从头讲一遍，而是只讲“它在这个项目里到底怎么用”。

### 7.1 当前有两个任务

你现在可以先记住：当前主流程里，最关键的是下面两个任务。

- `app_1ms_task`
- `app_10ms_task`

### 7.2 两个任务分别负责什么

你可以先把它们理解成“一个主任务，一个辅助任务”。

`app_1ms_task`

- 初始化系统硬件
- 初始化 WLC 协议栈
- 每 1ms 处理一次主状态机

`app_10ms_task`

- 周期调用 `tomato_calibration_mainfunction()`

### 7.3 一个很容易踩坑的点

以下内容根据源码结构整理补充

虽然第二个任务名字叫 `app_10ms_task`，但你千万不要只看名字就下结论。  
它当前源码里实际用的是：

```c
vTaskDelayUntil(&xLastWakeTime, pdMS_TO_TICKS(2));
```

也就是说，它现在真实运行周期是 `2ms`，不是名字暗示的 `10ms`。

这类地方对新手特别容易造成误导，所以这里给你一个非常实用的规则：

以后只要发现“函数名、任务名、变量名”和实际行为不一致，一律以源码真实逻辑为准，不要先相信名字。

### 7.4 为什么这里大量使用 `vTaskDelayUntil`

因为这个项目里有很多动作都依赖稳定节拍，而 `vTaskDelayUntil``更适合干这个。

- `vTaskDelay`：相对延时
- `vTaskDelayUntil`：绝对周期

你可以先这样理解：

- `vTaskDelay` 更像“我先睡一会儿”
- `vTaskDelayUntil` 更像“我每隔固定节拍准点执行一次”

在这种协议定时比较敏感的工程里，后者通常更稳。

## 9. 五层架构怎么理解

旧教程反复强调“这个项目是分层的”，这一点是对的，而且对新手尤其重要。  
因为只要你先把分层关系看清，后面就不会一上来把所有问题都混在一起。

你不用把“架构”这个词想得太大。  
这里说的五层，本质上只是回答一个问题：

`从硬件到协议逻辑，中间一共隔了哪几层？`

### 9.1 第一层：硬件

这一层就是最原始的硬件本体：

- 板子
- 线圈
- 驱动桥
- 采样电路
- ASK 检波
- MCU 本身

### 9.2 第二层：BSP 层

这一层负责直接碰硬件。

比如：

- 配 GPIO
- 开定时器
- 配 DMA
- 改 PWM 寄存器
- 测脉宽

### 9.3 第三层：WLC_driver 抽象接口层

这一层最值得记住的结构是 `wlc_lib_api_t`。

它把硬件能力抽象成函数指针表，例如：

- 读 ADC
- 设 PWM
- 使能 IRQ
- 调试打印

你可以先把它理解成一个“能力清单”：

- 上层只说“我要采样”“我要改 PWM”“我要开中断”
- 至于底层到底怎么做，由这层往下接

这样协议逻辑就不需要知道底层到底是 APM32，还是以后换成别的芯片。

### 9.4 第四层：WLC_server 服务层

这里放的是一些不属于某个单独协议阶段、但又反复会用到的通用能力：

- 软件定时器
- ASK 解调
- PWM 功率桥控制

### 9.5 第五层：WLC_app 应用层

这里是整个协议状态机最核心的一层。

真正知道下面这些事情的，主要就是这一层：

- 当前处于哪个 Qi 阶段
- 收到哪个包该怎么解释
- 什么时候进入下一阶段

## 10. `wlc_lib` 为什么这么重要

如果你后面只能先记住一个关键名字，那大概率就是 `wlc_lib`。  
因为它几乎就是“上层协议逻辑”和“底层硬件实现”之间的中间桥。

你可以先把 `wlc_lib` 理解成一个“翻译层”。

它的核心思路是：

```text
协议层调用抽象接口
-> 抽象接口映射到 bsp_xxx
-> bsp_xxx 再去操作硬件
```

这种设计对项目维护的好处非常直接：

1. 上层协议代码可复用
2. BSP 可以单独调试
3. 后续换芯片时，改动主要集中在 BSP 和映射表

以下内容根据源码结构整理补充

当前映射表是在 `Core/Src/freertos_task.c` 中初始化的，里面已经把这些能力绑定到了 BSP。  
如果你以后想追“某个抽象接口最后到底落到了哪个底层函数”，这一段映射关系就是重点。

- ADC 时钟 / 组切换 / 采样值读取
- PWM 时钟 / 使能 / 回调 / ARR / 脉宽 / 相位
- IRQ 使能
- UART 打印

## 11. BSP 层模块说明

这一章建议你带着一个问题去读：

`如果硬件相关的事情出了问题，我应该先去哪个文件找？`

下面每个模块都按“代码在哪、它是干什么的、你应该怎么理解”来讲。

### 11.1 UART 调试串口

位置：

- `Bsp/src/bsp_uart.c`
- `Bsp/inc/bsp_uart.h`

它主要做两件事：

- 提供调试日志输出
- 用环形缓冲 + DMA 提高发送效率

新手可以先这样理解：

- 上层不是直接 `printf` 到硬件
- 而是通过 `wlc_lib_uart_printf` 间接走到这里

### 11.2 Timer 时基

位置：

- `Bsp/src/bsp_timer.c`
- `Bsp/inc/bsp_timer.h`

它的核心作用是：

- 产生 1ms 周期时基
- 在中断中回调 `wlc_lib_timer_handle_int()`

你可以把它理解成整个协议栈的“公共时钟”。

### 11.3 IRQ / ASK 输入

位置：

- `Bsp/src/bsp_irq.c`
- `Bsp/inc/bsp_irq.h`

它的关键作用是：

- 从 `PB7` 捕获 ASK 相关边沿
- 计算相邻边沿时间差
- 把脉宽丢给 `wlc_lib_ask_handle_int()`

如果你发现 ASK 根本收不到包，或者收包时序明显不对，这里通常就是排查重点。

### 11.4 PWM 功率桥输出

位置：

- `Bsp/src/bsp_pwm.c`
- `Bsp/inc/bsp_pwm.h`

它主要负责：

- 驱动全桥 / 半桥输出
- 设置频率、相位和占空比
- 配合上层完成 Ping、功率传输和 FSK

### 11.5 ADC 采样

位置：

- `Bsp/src/bsp_adc.c`
- `Bsp/inc/bsp_adc.h`

### 11.8 为什么 `bsp_timer.c` 和 `bsp_irq.c` 更特殊

旧教程里专门强调过这一点，这个提醒非常值得保留。  
因为这两个文件和别的 BSP 文件不太一样。

大部分 BSP 文件只负责“执行硬件动作”，例如：

- UART 负责发日志
- PWM 负责改输出
- ADC 负责采样

但 `bsp_timer.c` 和 `bsp_irq.c` 不一样，它们还负责把硬件事件主动交给 WLC 协议栈：

- `bsp_timer.c` 在定时器中断里调用 `wlc_lib_timer_handle_int()`
- `bsp_irq.c` 在外部中断里调用 `wlc_lib_ask_handle_int()`

所以这两个文件需要 `#include "wlc_lib.h"`。  
原因不是“BSP 直接依赖应用层”，而是它们要把硬件中断事件继续转交给 WLC 协议栈。

可以把这个方向记成：

```text
硬件事件
-> BSP 中断服务
-> wlc_lib_xxx_handle_int()
-> WLC 协议栈继续处理
```

### 11.9 新手读 BSP 的推荐顺序

如果你准备第一次顺着 BSP 往下读，更适合新手的顺序是：

1. `ring_buffer` / UART
2. Timer
3. IRQ
4. PWM
5. ADC
6. QC / EEPROM

这样你会先建立“日志怎么出来、时钟怎么跑、ASK 怎么进来、功率怎么出去”的主线，再去看附属模块。

## 13. 应用层总状态机

这一章是整个项目里最核心的一章之一。  
如果你后面只能认真读几份文件，那 `wlc_app.c` 一定在里面。

应用层入口在：

- `User/WLC_app/src/wlc_app.c`

这是整个项目最核心的文件之一。

### 13.1 `wlc_context_t` 保存什么

每个设备都会有一个 `wlc_context_t`。  
你可以把它理解成“这个设备当前所有协议运行状态的大盒子”。  
里面会保存：

- 当前阶段
- 上一阶段
- 当前阶段超时定时器
- 最近接收的数据包
- ID 信息
- 配置信息
- 协商结果
- 功率传输合同
- 阶段初始化标志
- 输入电压状态

### 13.2 `wlc_app_task()` 做什么

它不是把所有业务逻辑都堆在一个大函数里，而是先看当前处于哪个阶段，再把处理分发到对应阶段函数。

这类写法对协议状态机特别友好，因为它：

- 清楚
- 可维护
- 好调试

### 13.3 `set_phase()` 为什么重要

这个函数非常重要，因为它不是简单“改一下阶段值”，而是会连同阶段切换要做的收尾和初始化一起处理。

它负责：

- 销毁旧阶段的超时定时器
- 切换到新阶段
- 清除 `phase_initialized`
- 创建新阶段超时定时器

所以以后只要遇到“为什么阶段切换后还有一堆别的动作”，优先回来看这个函数。

### 13.4 `phase_initialized` 是干什么的

它表示“当前阶段第一次进入时要做的初始化，已经做过没有”。

这样设计的好处很直接：

- 阶段只在第一次进入时做一次启动动作
- 后续每次循环只做持续处理

这样能避免同一个阶段在每 1ms 循环里重复初始化。

## 14. Qi 协议完整流程

从这一章开始，你可以把注意力从“工程怎么组织”切换到“协议到底怎么一步一步往前跑”。  
对第一次接触这个项目的人来说，最重要的不是一开始就记住每个包头，而是先把大流程记住。

当前工程的主阶段顺序是：

```text
Selection
-> Ping
-> ID/Config
-> Negotiation（仅 EPP）
-> Power Transfer
```

其中：

- `BPP`：通常在 `ID/Config` 后直接进 `Power Transfer`
- `EPP`：在 `ID/Config` 后先进入 `Negotiation`

你现在可以先把这几个阶段记成一句人话：

- `Selection`：收拾现场，准备重新开始
- `Ping`：看看附近是不是有设备
- `ID/Config`：确认设备是谁、支持什么
- `Negotiation`：只有 EPP 才需要继续谈扩展能力
- `Power Transfer`：正式进入闭环充电

下面再按阶段分别讲。

---

## 15. Selection 阶段

### 15.1 这个阶段做什么

Selection 阶段在当前工程里，更像“协议入口”和“失败后退回来的复位点”。

你可以先把它理解成：先把上一次流程残留的东西清一遍，再准备重新开始。

它主要做这些事：

- 停止上一个阶段遗留的动作
- 清空部分上下文
- 关闭 ASK
- 准备切到 Ping

### 15.2 它为什么很重要

虽然这个阶段本身逻辑不复杂，但它很重要。  
因为很多失败路径最后都会退回到这里。

所以如果你调试时发现系统总是回到 Selection，通常不是 Selection 本身有问题，而是前面某个阶段失败后退回来了。  
这时可以先往下面这些方向想：

- 某阶段超时
- 收到 EPT
- 协商失败
- 功率传输失败

---

## 16. Ping 阶段

### 16.1 Ping 是做什么的

Ping 的目标不是马上开始正式充电，而是先确认一件最基础的事：

`线圈附近是不是有符合预期的接收设备`

### 16.2 这个阶段怎么工作

这一步你可以先理解成“发射端在试着喊话，看附近有没有合规的接收端回应”。

发射端会：

- 开启 ASK 接收
- 设置 Ping 对应功率
- 发送固定周期的 Ping 脉冲
- 等待手机发回 `SIG` 包

### 16.3 代码位置

- `User/WLC_app/src/Qi_Ping.c`

### 16.4 成功后会发生什么

一旦 `qi_ping_process()` 判断检测成功：

- 停止 Ping
- 保持当前功率输出不断
- 进入 `ID/Config`

这里“停止 Ping，但保持当前功率不断”非常关键。  
因为后续阶段还要继续基于当前链路通信，不是识别到了设备就把输出全停掉。

### 16.5 新手怎么理解 SIG 包

如果你以前没见过这个包，可以先最简单地记成：

`SIG = 手机在 Ping 阶段告诉发射端：我在这里`

## 17. ID/Config 阶段

### 17.1 这个阶段的本质

Ping 只能知道“附近有设备”，但还不知道：

- 它到底是什么设备
- 它支持什么模式
- 它最多想要多大功率
- 后面是走 `BPP` 还是 `EPP`

ID/Config 阶段就是解决这个问题的。

### 17.2 这一阶段会拿到什么信息

这一阶段常见会拿到这些信息：

- Qi 版本
- 厂商代码
- 基础设备 ID
- 是否有扩展 ID
- 最大功率
- 控制窗口大小
- 控制窗口偏移
- 是否支持 EPP 协商

### 17.3 关键包

- `ID`
- `XID`（可选）
- `PCH`（可选）
- `CFG`

### 17.4 代码位置

- `User/WLC_app/src/Qi_IdConfig.c`

### 17.5 阶段结束后的分流

这一阶段结束后，不会无脑只走一条路，而是会分流。  
当前源码逻辑是：

- 如果解析结果是 `BPP`
  - 进入 `Power Transfer`
- 如果解析结果是 `EPP`
  - 进入 `Negotiation`

以下内容根据源码结构整理补充

这里需要按当前代码实现再加一句更准确的说明：

- `BPP` 路径在 `wlc_app.c` 里会直接切阶段到 `QI_PHASE_POWER_TRANSFER`
- 但当前文件里原本用于调用 `qi_pt_init()` / `qi_pt_start()` 的那段代码是注释掉的
- `EPP` 路径则是在 `Qi_Negotiation.c` 中，协商完成后真实调用了 `qi_pt_init(device, QI_PHASE_CALIBRATION, ...)` 和 `qi_pt_start(device)`

所以从“当前代码行为”看：

- `EPP` 进入 PT 的初始化链路是明确存在的
- `BPP` 进入 PT 的阶段切换是明确存在的
- 但 `BPP` 的 PT 模块初始化调用，在 `wlc_app.c` 当前版本里并没有像 `EPP` 那样直接看到启用代码

这类地方以后要一律以源码现状为准，不要直接沿用旧 HTML 的理想流程图。

### 17.6 这一步内部其实还有一个小状态机

旧教程里把这一段讲得很细，这部分应该保留下来。  
因为它能直接解释一个新手很容易困惑的问题：

`为什么有时收包顺序变了，代码还是能接住？`

这个小状态机的常见状态有：

- `WAIT_ID`
- `WAIT_XID`
- `WAIT_OPTIONAL`
- `WAIT_CFG`
- `COMPLETE`
- `ERROR`

可以先这样理解每个状态：

- `WAIT_ID`：先等第一个 `ID`
- `WAIT_XID`：如果 `ID` 说后面还有扩展身份，就继续等 `XID`
- `WAIT_OPTIONAL`：开始接可选包，例如 `PCH` 或部分厂商私有包
- `WAIT_CFG`：最后必须收 `CFG`
- `COMPLETE`：这一阶段收齐了
- `ERROR`：收到不该来的包，或关键字段校验失败

一个很实用的状态转换速记是：

```text
WAIT_ID
-> WAIT_XID        （ID 里 ext_bit = 1）
-> WAIT_OPTIONAL   （ID 里 ext_bit = 0）
-> WAIT_CFG        （遇到 CFG 时切入）
-> COMPLETE        （CFG 解析成功）
```

### 17.7 为什么它会“递归再处理一次”

这里是旧教程里最值得保留的细节之一。  
因为它说明当前实现不是“死板地按理想顺序等包”，而是会尽量接住提前到来的合法包。

例如当前状态在 `WAIT_XID`，按理说应该等 `XID`，但如果实际收到的是 `PCH`：

- 这个包不是 `XID`
- 但它也不是错误包
- 它其实是后面 `WAIT_OPTIONAL` 阶段允许出现的合法包

所以更合理的处理方式不是丢掉，而是：

1. 先把状态切到 `WAIT_OPTIONAL`
2. 再让同一个包按 `WAIT_OPTIONAL` 规则重新走一遍

同理，`WAIT_OPTIONAL` 收到 `CFG` 时，也会先切到 `WAIT_CFG`，再按 `CFG` 规则继续处理。

这就是为什么旧教程会强调：

`WAIT_XID -> WAIT_OPTIONAL -> WAIT_CFG`

这条链路里“递归处理”不是炫技，而是在避免合法包被错误丢弃。

### 17.8 `ID` 包里最值得看的字段

`ID` 包承担的是第一轮身份识别任务。

最关键的字段可以这样记：

- `byte[0]`：协议版本
- `byte[1..2]`：厂商码
- `byte[3]`：包含 `ext_bit` 和 `basic_id` 高位
- `byte[3..5]`：可以拼出基础设备 ID

新手读这一步时，最容易漏掉的是 `ext_bit`。

它的意义非常直接：

- `ext_bit = 1`：后面还应该有 `XID`
- `ext_bit = 0`：可以直接跳过 `XID`

所以如果你看到代码总停在 `WAIT_XID`，首先别怀疑手机，一定先检查：

- `ID` 包有没有解调错位
- `ext_bit` 有没有解析反

### 17.9 `CFG` 包里最关键的字段

`CFG` 是这个阶段里最能决定“后面往哪走”的包。

重点字段建议这样记：

- `reference_power`：参考功率，单位是 `0.5W`
- `count`：前面应该出现几个可选包
- `window_size`：功率调节窗口大小
- `window_offset`：窗口偏移
- `neg_bit`：是否进入 `EPP`
- `depth`：FSK 调制深度

其中最影响流程的有 3 个字段：

`reference_power`

- 例如 `10` 可理解成 `5W`
- 例如 `30` 可理解成 `15W`

`count`

- 它必须和前面实际收到的可选包数量对得上
- 如果对不上，通常说明中途丢包了

`neg_bit`

- `0`：按 `BPP` 走
- `1`：按 `EPP` 走

以下内容根据源码结构整理补充

这里有一个必须按当前代码说明清楚的细节：

- `qi_cfg_data_t` 的字段注释里，`window_size` / `window_offset` 写的是“单位 4ms”
- 但 `Qi_IdConfig.c` 在解析 `CFG` 时，已经执行了 `* 4`
- 也就是说，`ctx->cfg_data.window_size` 和 `ctx->cfg_data.window_offset` 在当前运行时更接近“已经换算过的毫秒值”

同时，`wlc_app.c` 在给 `ctx->contract.window_size_ms` 和 `ctx->contract.window_offset_ms` 赋值时，又写了一次 `* 4`。

因此教程里不能把这块简单写成“所有地方都还是 4ms 单位”。

更准确的说法应该是：

- `qi_cfg_data_t` 的类型注释和当前解析代码之间存在单位口径差异
- 当前代码运行路径里，`cfg_data` 已经做过一次换算
- `contract` 赋值处又做了一次乘 `4`

这属于当前源码里需要特别留意的实现细节，不适合直接沿用旧 HTML 的单位描述。

### 17.10 这一阶段常见错误怎么理解

这一阶段最常见的失败，很多时候不是“协议算法错了”，而是“收包链路某个环节歪了”。

常见现象和对应思路如下：

- 一直停在 `WAIT_ID`：先查 ASK 解调是否真的收到了 `ID`
- `WAIT_ID` 直接报错：说明第一个有效包不是 `ID`
- 卡在 `WAIT_XID`：重点查 `ext_bit`
- `INVALID_COUNT`：重点查可选包数量统计和 `CFG.count`
- EPP 设备被识别成 BPP：重点查 `CFG` 里的 `neg_bit`

## 18. Negotiation 阶段

这一章只和 `EPP` 路径强相关。  
如果你后面确认自己的设备一直按 `BPP` 跑，那这章的重要性会低一些；  
但如果你要做 `15W` 或更高扩展功率相关功能，这一章就很关键。

### 18.1 为什么会有这个阶段

并不是所有 Qi 设备都要走 Negotiation。  
这个阶段主要服务于 `EPP` 扩展功率场景。

### 18.2 它做什么

它负责继续把扩展能力和协商参数谈清楚，而不是重复做一遍设备识别。

你可以把它理解成：

`ID/Config 先知道“你是谁、你大概要什么”，Negotiation 再把扩展能力谈清楚`

### 18.3 常见处理内容

- `GRQ`
- `FOD`
- `WPID`
- `SRQ` 各类子命令

### 18.4 代码位置

- `User/WLC_app/src/Qi_Negotiation.c`

### 18.5 结束条件

协商完成后，代码会设置结束标志，再进入 `Power Transfer`。

### 18.6 `GRQ / SRQ / FOD / WPID` 可以先这样分工

旧教程里这几类包拆得比较细。  
统一教程里不一定要把所有细节一口气塞给新手，但至少要先把它们分别是干什么的讲清楚。

`GRQ`

- 是“通用请求”
- 接收端用它向发射端要通用资料
- 最常见的是：
  - 请求 `PTx_ID`
  - 请求 `CAP`

`SRQ`

- 是“具体请求”
- Negotiation 阶段里最重要的一类协商包
- 接收端会用它提出更具体的参数请求

`FOD`

- 是接收端发来的异物检测参考信息
- 常见可理解为“我这边用什么基准看 FOD”

`WPID`

- 是接收端无线电力 ID
- 更像“接收端型号身份补充信息”

### 18.7 发射端在 Negotiation 阶段通常会怎么回

旧教程里把发射端响应方式也讲得很细，这部分非常值得保留。  
因为新手常常只盯着“收到了什么包”，却忘了还要看“发射端回了什么”。

常见响应可以先这样记：

- `ACK`：收到并接受
- `NAK`：收到但拒绝
- `ND`：知道了，但没有额外数据返回
- `PTx_ID`：返回发射端身份
- `CAP`：返回发射端能力

最常见的对应关系是：

- `GRQ(0x30)` -> 返回 `PTx_ID`
- `GRQ(0x31)` -> 返回 `CAP`
- `FOD` 中某些参考值 -> 可能回 `ACK` 或 `ND`
- `WPID` -> 通常按“接收并确认”来处理

### 18.8 `SRQ` 为什么麻烦

因为它不是“一个包只代表一个固定意思”，而是一类“带子命令的请求”。

旧教程里给出的核心思路是：

- `SRQ` 的 `byte[0]` 更像子命令号
- `byte[1]` 往往是这个子命令对应的参数

你现在不需要把每个子命令全部背下来，但至少要知道：

- 协商阶段很多细参数，不是靠 `GRQ`，而是靠 `SRQ`
- `SRQ` 处理完后，通常还会影响协商内部计数和是否进入下一阶段

### 18.9 为什么旧教程反复强调 `change_count`

这是 Negotiation 里很关键、但又很容易被新手忽略的机制。

可以把它理解成：

- 手机说自己一共改了几次协商参数
- 发射端也自己记一份“我收到并接受了几次”
- 最后结束协商时，两边要对账

如果对不上，就说明：

- 中间可能丢包了
- 或者某次协商请求没有被正确处理

所以 `change_count` 本质上是在避免：

`手机以为参数已经改好，但发射端其实少收了一步`

### 18.10 一个很重要的实现习惯

旧教程里还有一个很实用的提醒，这里也要保留：

- Negotiation 里的包不一定严格按固定顺序来
- 代码很多时候更像“按包类型分别处理”
- 每次 FSK 响应发完，再通过回调决定是继续等包，还是进入下一阶段

所以你以后排查 Negotiation，不要先假设它一定是死板线性流程，而是要看：

- 当前收到了什么包
- 回了什么响应
- `change_count` 有没有对上
- 回调之后是不是准备切下一阶段了

以下内容根据源码结构整理补充

按当前 `Qi_Negotiation.c` 的实现，协商成功进入下一阶段时，代码会：

1. 设置 `ctx->neg_data.end_negotiation_flag = true`
2. 直接调用 `qi_pt_init(device, QI_PHASE_CALIBRATION, ...)`
3. 再调用 `qi_pt_start(device)`

这说明当前实现里的 `EPP` 协商不是“只改一个标志，等别处慢慢接手”，而是已经在协商模块内把 PT 校准阶段的初始化做掉了。

## 19. Power Transfer 阶段

这是整个项目里最“像真的在充电”的阶段。  
前面那些阶段更像准备、识别、确认、协商；到了这里，才是真正进入闭环功率传输。

### 19.1 这是最核心的充电阶段

到了这里，才算正式进入闭环充电。

### 19.2 这个阶段处理什么

这一阶段最重要、最常碰到的包包括：

- `CE`
- `RP8`
- `RP`
- `CHS`
- `PCH`
- `EPT`

### 19.3 每种包可以怎么理解

你可以先按“它们分别在表达什么”来理解。

`CE`

- 手机告诉发射端：功率偏高了还是偏低了
- 发射端据此调节输出

`RP8 / RP`

- 手机报告自己接收到的功率
- 发射端据此做统计、校准、FOD 判断

`PCH`

- 控制调功延时参数

`CHS`

- 充电状态类信息

`EPT`

- 表示结束传输，或因错误、过温、重协商等原因退出

### 19.4 代码位置

- `User/WLC_app/src/Qi_Power_Transfer.c`
- `User/WLC_app/src/Qi_Power_Delivery.c`

### 19.5 一个非常重要的实现细节

以下内容根据源码结构整理补充

虽然类型定义里有 `QI_PHASE_CALIBRATION`，但按当前顶层应用状态机的分发表来看，真正被顶层直接 dispatch 的阶段是：

- `Selection`
- `Ping`
- `ID/Config`
- `Negotiation`
- `Power Transfer`

而 `Calibration` 在当前实现里更像 `Power Transfer` 内部的子阶段，不是顶层独立 dispatch 阶段。

这个区别在阅读教程和阅读源码时一定要分清。  
不然你很容易误以为 `Calibration` 和 `Ping`、`ID/Config` 一样，是顶层并列阶段。

### 19.6 `BPP` 和 `EPP` 进入 PT 的路径不一样

虽然最后都会走到 Power Transfer，但进入方式并不一样。  
这个差别对调试很重要。

`BPP` 路径：

- `ID/Config` 完成
- `CFG.neg_bit = 0`
- 直接进入正式功率传输
- 主要处理 `CE + RP8 + CHS + EPT`

`EPP` 路径：

- `Negotiation` 完成
- `CFG.neg_bit = 1`
- 先进入校准逻辑
- 之后再进入正式功率传输
- 主要处理 `CE + RP + CHS + EPT`

所以调试时不要只看“有没有进 PT”，还要继续问：

- 是直接进正式充电
- 还是先走了校准子流程

以下内容根据源码结构整理补充

这一段必须再补一条“按当前实现”的说明：

- `EPP` 路径的 `qi_pt_init/qi_pt_start` 调用，在 `Qi_Negotiation.c` 中能直接看到
- `BPP` 路径在 `wlc_app.c` 里当前只明确看到了阶段切换
- 而 `wlc_app.c` 里那段本应初始化 PT 的直接调用代码目前是注释状态

所以教程如果要完全忠于源码，不能简单写成“BPP 和 EPP 只是进入 PT 的 phase 参数不同”，因为当前实现并不完全对称。

### 19.7 EPP 为什么要先校准

EPP 下，发射端不仅要“把电送过去”，还要尽量知道：

- 自己发出了多少功率
- 对方真正收到了多少功率

这两个数之间的差值，就是后面做 `pFOD` 判断的重要依据。

旧教程把 EPP 校准拆成 4 类 `RP` 类型，这个信息非常关键：

- `type1`：轻载校准点
- `type2`：重载校准点
- `type0`：参考功率点
- `type4`：最终功率点

新手可以先记住下面这个顺序：

```text
type1
-> type2
-> type0
-> type4
-> 转入正式 PT
```

以下内容根据源码结构整理补充

当前 `Qi_Power_Transfer.c` 的真实处理口径更适合这样描述：

- `packet->data[0]` 的低 3 位用来区分 `type0 / type1 / type2 / type4`
- `packet->data[1..2]` 被拼成当前实现里的功率原始值

所以这里虽然旧教程常把它叫成“24 位 RP”或“高位 RP 报告”，但按当前代码实现去讲，更稳妥的说法是：

- `RP` 包是 3 字节负载
- 其中 1 字节偏类型控制
- 另外 2 字节参与功率值计算

### 19.8 `CE` 包不是收到就立刻调功

收到 `CE` 后，发射端通常不会立刻马上去改 PWM，而是会先进入一个 `Holdoff` 时间窗口。

目的很简单：

- 刚收到 `CE` 时，磁场和采样值可能还不稳定
- 立刻调功容易把反馈搞得更抖

所以会有一个典型时间线：

1. 收到 `CE`
2. 暂停部分 ASK 解调处理
3. 等 `holdoff`
4. 再执行功率调节
5. 再等一小段生效时间
6. 然后恢复正常处理

旧教程里给出的默认理解是：

- `holdoff` 默认大约 `10ms`
- `CE` 超时大约 `1828ms`

所以如果你发现“某些包偶尔像被跳过”，不要第一反应就怀疑解调器坏了。  
先确认是不是正处在 `CE Holdoff` 窗口里。

以下内容根据源码结构整理补充

当前 `Qi_Power_Transfer.c` 里的实际动作更接近：

- 收到 `CE` 后先把 `control_error_packet_flag` 置位
- 计算 `power_control_timeout = holdoff + QI_ACTIVE_TIME - 1`
- 在 `holdoff` 到点时才真正执行 `handle_ce_packet()`
- 在整个等待窗口里，ASK 接收会被临时关掉，结束后再恢复

另外还有一个很容易被旧教程带偏的点：

- `wlc_app.c` 里会先把 `ctx->contract.control_holdoff_ms` 设成 `QI_PCH_DEFAULT`
- 但 `qi_pt_init()` 当前真正写进 PT 模块的是 `QI_PCH_MIN`
- 而 PT 阶段当前的 `handle_pch_packet()` 只统计了 `PCH` 包次数，没有把收到的值直接写回 `power_control_holdoff`

所以如果你严格按现有代码看，`holdoff` 这条链路目前并不是“收到 PCH 后立即动态更新”的完整实现。

### 19.9 `RP8` 和 `RP` 的区别

这两个包名字很像，所以新手很容易混。  
但它们的用途层次并不一样。

`RP8`

- 常见于 `BPP`
- 数据更短
- 主要用来表达接收功率的粗粒度信息

`RP`

- 常见于 `EPP`
- 精度更高
- 不只是“报个数”，还会参与校准和 `pFOD`

所以排查时可以直接这样记：

- 看见 `RP8`，多半先往 `BPP` 思路走
- 看见高精度 `RP`，多半已经在 `EPP` 的更深层逻辑里了

以下内容根据源码结构整理补充

当前代码里还有一个实现级细节值得记住：

- `RP8` 到来时，会直接走 `handle_rp8_packet()`，并刷新 `QI_T_RP_TIMEOUT_TYP`
- `RP` 到来时，会先暂停接收，再调用更复杂的解析流程，并可能触发校准或 `pFOD` 逻辑
- 当前 `RP` 解析时实际取的是 `data[1]` 和 `data[2]` 组合值，再结合 `data[0]` 的低 3 位判断类型

### 19.10 `EPT / CHS / PCH / FOD` 分别怎么理解

`EPT`

- 表示结束功率传输
- 可能是正常充满，也可能是过温、过流、NFC 干扰、内部错误等原因

`CHS`

- 更像状态上报
- 常拿来表示充电状态或电量相关信息

`PCH`

- 协议语义上，它和功率控制延时参数有关
- 但按当前 `Qi_Power_Transfer.c` 实现，`handle_pch_packet()` 里只统计了包次数，还没有把收到的值直接写回 `power_control_holdoff`

`FOD`

- 表示与异物检测相关的信息
- 既可能来自接收端上报，也可能配合发射端自己的 `pFOD` 判断一起使用

### 19.11 `EPT` 原因码至少要认识这些

旧教程里给了比较完整的速查表。  
对新手来说，不用一口气背完全部原因码，但下面这些最好先认识。

- `0x01`：充电完成
- `0x02`：内部故障
- `0x03`：过温
- `0x04`：过压
- `0x05`：过流
- `0x08`：无响应
- `0x0B`：请求重启
- `0x0D`：RFID / NFC 干扰

你可以先把它们分成两类来记：

正常结束：

- `0x01`

异常结束：

- 过温、过压、过流
- 无响应
- 内部故障
- RFID / NFC 干扰

如果现场测试里总是掉 PT，第一时间要把 `EPT` 原因码打出来，不然你只能看到“退回了”，但不知道为什么退回。

### 19.12 正式 PT 和校准 PT 对 `RP` 的处理不完全一样

这一点旧教程讲得也很有价值。

在校准模式里：

- `type1 / type2` 更偏向建立校准点
- `type0 / type4` 更偏向参考点、最终点和 `ploss` 相关判断

在正式 PT 里：

- 某些校准型 `RP` 不再按校准方式处理
- 更关注当前功率报告和 `pFOD`

所以同样看到 `RP`，也要先问自己：

- 现在是在校准子阶段
- 还是已经进正式功率传输了

以下内容根据源码结构整理补充

按当前实现可以更具体地说：

- 校准阶段下，`type1` 和 `type2` 会尝试记录校准点，并按结果回 `ACK` 或 `NAK`
- 校准阶段下，`type0` 会记录参考点、做 `pFOD` 判断，并切到正式 PT
- 校准阶段下，`type4` 会记录最终点、做 `pFOD` 判断，并切到正式 PT
- 正式 PT 下，`type0` 仍可能回 `ACK`
- 正式 PT 下，`type1 / type2` 不再按校准点处理，而是走 `ND` 响应

### 19.13 ASK 解调为什么会“故意暂停”

旧教程里专门提到过这一点，这能解释很多“看起来像丢包”的现象。  
这类问题非常容易把新手带偏，所以这里一定要保留。

典型情况包括：

- 收到 `CE`，进入 `Holdoff`
- 收到较复杂的 `RP`，处理中暂时停解调
- 某些有效包刚收到，等当前处理和响应结束再恢复

所以有时不是解调器坏了，而是代码为了控制时序，主动让它短暂停一下。

### 19.14 FSK 响应发完后的回调也很重要

旧教程里提到 `qi_pt_response_callback()` 这类回调，它的价值在于：

- FSK 响应不是“写个变量就算发完”
- 真正发完之后，系统还要做收尾
- 一个常见动作就是恢复 ASK 解调

这也是为什么有些问题不能只看“准备发了什么”，还要看：

- 发出去没有
- 发完后的回调有没有执行

以下内容根据源码结构整理补充

当前 `qi_pt_response_callback()` 里最直接能确认的动作，是重新打开 IRQ 接收：

- `wlc_lib_irq_enable(0, 1)`

所以它的一个核心作用，就是把“刚才为了处理和响应而暂停的接收链路”恢复回来。

### 19.15 PT 阶段更细的速查

如果你只想先快速回忆这一阶段，记下面这张脑内表就够用了：

- `CE`：闭环调功
- `PCH`：和 `holdoff` 相关，但当前 PT 实现里未直接改写运行中的 `holdoff`
- `RP8`：BPP 功率报告
- `RP`：EPP 功率报告 / 校准 / `pFOD`
- `CHS`：状态上报
- `FOD`：异物检测相关信息
- `EPT`：结束并给原因

### 19.16 `qi_pt_process()` 更适合按“当前源码执行顺序”理解

如果你对着 `Qi_Power_Transfer.c` 直接看，当前更适合按下面这个执行顺序理解：

1. 如果 FSK 响应还在发送，就先等待响应结束
2. 如果前面已经收到了 `CE`，就先处理 `CE Holdoff` 这条延迟控制链
3. 如果此时有新包，再按包头分发处理，这一步里面才会遇到 `EPT / RP / RP8 / CHS / PCH / FOD`
4. 如果没有新包，再去做 `RP` 超时和 `CE` 超时的递减检查

以下内容根据源码结构整理补充

这和很多教程喜欢画的“先看是否结束、再看新包、最后看后台任务”的理想流程图不完全一样。  
当前代码里，`EPT` 并不是函数一进来就优先单独处理，而是在“收到新包”这个分支里，按 `header` 分发时处理到的。

另外还有一个和当前实现强相关的点：

- `qi_pt_process()` 一开始就要求 `pt->active == true`
- 如果 PT 模块没有先被 `qi_pt_init()` 和 `qi_pt_start()` 激活，它会直接返回 `false`
- `wlc_app.c` 在收到这个 `false` 时，不会立刻退出 PT，而是会继续刷新顶层 `QI_PHASE_POWER_TRANSFER` 的超时

所以调试 PT 时，不能只看“phase 有没有切到 PT”，还要确认“PT 模块自己有没有真的 active”。

### 19.17 超时和 `pFOD` 是 PT 的两条安全线

旧教程里有两个数字很值得先记住：

- `CE` 超时约 `1828ms`
- `RP` 超时典型约 `23000ms`

而在校准阶段，`RP` 超时通常会更短。

这类超时的核心意思可以先理解成：

- 太久收不到 `CE`，闭环调功已经不可信
- 太久收不到 `RP / RP8`，接收端功率报告链路也不可信

除了超时，另一个安全机制是 `pFOD`。

它的核心思想是：

`发射功率 - 接收功率 = 功率损耗`

如果这个损耗连续多次超出阈值，就要怀疑有异物在吸能发热。

旧教程里特别强调：

- 不是一次超阈值就立刻判死
- 而是连续计数
- 连续达到阈值次数后才触发

这也是为了避免误报。

## 20. 数据包怎么读

这一章是专门写给“看见 `header`、`data[]`、`checksum` 就有点发怵”的人的。  
你不用先把所有包都背下来，只要先学会按顺序看，一个数据包就没有那么难。

项目里最核心的协议结构体之一是 `qi_packet_t`。

你现在先知道它里面大概装了这些东西就够了：

- `header`
- `data[27]`
- `length`
- `valid`

### 20.1 先看 ASK 完整帧长什么样

一个完整的 ASK 帧通常可以先理解成：

```text
前导码 + Header + Data + Checksum
```

更具体一点：

- 前导码：`11` 到 `25` 个 bit `1`
- `Header`：`1` 字节
- `Data`：`0` 到 `27` 字节
- `Checksum`：`1` 字节

### 20.2 为什么一个字节在链路上看起来不是 8 位

Qi 的 ASK 字节不是“裸 8 位”，而是 11 位格式：

```text
起始位(0)
+ D0 D1 D2 D3 D4 D5 D6 D7
+ 奇偶校验位
+ 停止位(1)
```

这里最容易踩坑的是：

- 数据位是 `LSB first`
- 也就是最低位先发

### 20.3 常见脉宽阈值怎么记

旧教程给了很实用的经验值：

- `bit 1`：约 `130-340us`
- `bit 0`：约 `370-630us`

你可以先把它粗略记成：

- 短脉冲更像 `1`
- 长脉冲更像 `0`

### 20.4 校验和怎么理解

这里用的是异或校验思路。

可以先简单记成：

```text
checksum = Header XOR Data[0] XOR Data[1] XOR ... XOR Data[N]
```

### 20.5 阅读顺序建议

不要上来就盯着 `data[]` 看。  
正确顺序是：

1. 先看当前处于哪个阶段
2. 再看 `header` 是什么包
3. 最后按该包规则解释 `data[]`

### 20.6 常见包头速查

| 包头 | 名称 | 常见阶段 |
| --- | --- | --- |
| `0x01` | `SIG` | Ping |
| `0x02` | `EPT` | Power Transfer |
| `0x03` | `CE` | Power Transfer |
| `0x04` | `RP8` | Power Transfer |
| `0x06` | `PCH` | ID/Config / PT |
| `0x07` | `GRQ` | Negotiation |
| `0x20` | `SRQ` | Negotiation |
| `0x22` | `FOD` | Negotiation / PT |
| `0x31` | `RP` | Power Transfer |
| `0x51` | `CFG` | ID/Config |
| `0x71` | `ID` | ID/Config |
| `0x81` | `XID` | ID/Config |

### 20.7 关键包一句话速记

- `ID`：我是谁
- `XID`：我的扩展身份
- `PCH`：协议上是调功延时相关参数；当前 PT 实现里还没有把收到的值直接写回运行中的 `holdoff`
- `CFG`：我支持多大功率、走 BPP 还是 EPP
- `SIG`：我在这里，Ping 能收到我
- `CE`：你功率偏高了或偏低了，快调
- `RP8`：我现在收到了多少功率，BPP 常用
- `RP`：我现在收到了多少功率，EPP 常用而且更精细
- `EPT`：我要结束传输
- `CHS`：我现在是什么充电状态
- `GRQ`：我想要你给我通用信息
- `SRQ`：我想要你给我特定信息
- `FOD`：和异物检测相关
- `WPID`：和接收端扩展能力请求相关
- `PTx_ID`：发射端把自己的身份告诉接收端
- `CAP`：发射端把自己的能力告诉接收端
- `NEG`：进入或触发重新协商相关流程

### 20.8 旧教程里更细的几个包，至少要先记住用途

`GRQ`

- 用来请求发射端通用信息
- 最常见是要 `PTx_ID` 和 `CAP`

`SRQ`

- 用来发具体协商请求
- Negotiation 阶段很核心

`FOD`

- 用来携带异物检测相关参考信息或状态

`WPID`

- 用来携带接收端无线电力 ID

`PTx_ID`

- 发射端的身份包
- 常作为 `GRQ(0x30)` 的回复

`CAP`

- 发射端能力包
- 常作为 `GRQ(0x31)` 的回复

### 20.9 `EPT` 原因码速记版

为了减少来回翻文档，这里把最常见的结束原因再压缩成一张速记表：

| 代码 | 含义 | 你应该先想到什么 |
| --- | --- | --- |
| `0x01` | 充电完成 | 正常结束 |
| `0x02` | 内部故障 | 接收端或链路异常 |
| `0x03` | 过温 | 温升或散热问题 |
| `0x04` | 过压 | 输入或调功异常 |
| `0x05` | 过流 | 功率链路异常 |
| `0x08` | 无响应 | 通信链路丢失 |
| `0x0B` | 重启 | 接收端要求重新开始 |
| `0x0D` | RFID/NFC 干扰 | 近场卡片或干扰源导致退出 |

### 21.4 Keil 构建时常见报错怎么查

以下内容根据源码结构整理补充

旧教程里有一部分“编译报错对照表”对新手非常有用，但里面混有历史版本路径。这里按当前工程结构把真正还能用的排查思路整理出来：

| 现象 | 优先检查什么 |
| --- | --- |
| `No section matches ... RESET` | 启动文件没有加入工程。当前工程应检查 `Drivers/Device/Geehy/APM32F10x/Source/arm/startup_apm32f10x_hd.s` 是否在 Keil 分组里。 |
| `multiply defined (SVC/PendSV/SysTick)` | FreeRTOS 和中断模板重复定义了异常处理函数。优先检查 [apm32f10x_int.c](/D:/apm32_wlc_code/wlc_code/Core/Src/apm32f10x_int.c) 里是否还保留了和 FreeRTOS 冲突的实现。 |
| `DMAx_IRQHandler multiply defined` | 两个源文件定义了同一个 DMA 中断函数。优先检查 BSP 串口或 DMA 相关文件有没有重复 ISR。 |
| 找不到 `tomato_calibration_mainfunction` 一类符号 | 先检查 `Lib/Tomato_wlc.lib` 是否已加入 Keil 工程，再检查库搜索路径和链接顺序。 |
| 打开工程后直接一串器件相关错误 | 优先检查 Device Pack 是否完整，芯片是否仍是 `APM32F103CB`。 |

对新手来说，一个很实用的习惯是：

1. 先确认 `.uvprojx` 能正常打开
2. 再确认启动文件、库文件、BSP 源文件都在工程分组里
3. 再看具体报错属于“缺文件”“重复定义”还是“器件环境不对”

这样比上来就盯着第一条报错抠细节更稳。

## 23. 常见错误排查

这一章可以理解成“现场出问题时的第一轮排查清单”。  
新手最容易犯的错，就是一遇到异常就同时怀疑十几个地方。  
更稳的做法，是先根据表象判断问题更像落在哪一层。

### 23.1 手机放上去完全没反应

这类现象说明最前面的链路可能就没起来。  
先别急着看复杂协议，优先检查最基础的运行条件：

优先检查：

- PWM 有没有输出
- `app_1ms_task` 有没有运行
- ASK IRQ 有没有进中断
- 串口有没有日志

### 23.2 总卡在 Ping

如果总卡在 Ping，说明“发射端已经开始找设备了”，但“接收端回应没有被正确接住”。

优先检查：

- 线圈是否真的发出 Ping
- `SIG` 包是否被正确解调
- `bsp_irq.c` 脉宽计算是否正常
- ASK 输入链路是否通

### 23.3 能进 ID/Config，但后面走不下去

这说明 Ping 阶段大概率已经没问题了，问题开始收敛到识别配置和后续分流。

优先检查：

- `ID / CFG` 包是否解析成功
- `BPP / EPP` 判断是否符合实际设备
- 协商阶段是否超时

### 23.4 充电时反复掉回 Selection

这类现象通常说明“前面能走通，但后面维持不住”。  
也就是说，问题更像发生在正式通信、调功、功率报告或异常退出这条链路上。

优先检查：

- 是否收到 `EPT`
- 是否 `CE / RP` 超时
- FOD 是否误判
- 输入电压是否异常

### 23.5 某个模块有文件，但感觉没在跑

以下内容根据源码结构整理补充

这是当前工程里真实存在的情况。  
新手最容易在这里踩坑，因为看到文件在仓库里，就下意识觉得它一定已经接进主流程了。

例如：

- `EEPROM` 代码存在
- 但当前启动流程没有看到它被接入

所以以后判断“模块有没有启用”，一定要看：

- 是否被初始化
- 是否被上层调用
- 是否进入任务循环

不能只看“文件在不在”。

### 23.6 任务、定时器中断、ASK 中断到底谁在做什么

以下内容根据源码结构整理补充

这个问题很适合新手尽早搞清楚。  
不然后面你很容易把“1ms 主循环”和“1ms 定时中断”混成一件事。

- `wlc_lib_handler()`：在 `app_1ms_task` 里周期调用，用来推进协议栈和应用层状态机
- `wlc_lib_timer_handle_int()`：由定时器中断回调触发，用来推进协议栈用到的时间基准
- `wlc_lib_ask_handle_int()`：由 ASK 相关 IRQ 回调触发，用来把脉宽输入送进解调链路

你可以先把它们理解成三条不同职责的执行流：

1. FreeRTOS 任务流：负责“每 1ms 跑一次主逻辑”
2. 定时器中断流：负责“给协议栈提供稳定 tick”
3. ASK 外部中断流：负责“有边沿就立刻测脉宽并解调”

它们不是同一个入口，只是最后都会服务同一个协议栈。

### 23.7 为什么 `goto_selection()` 不能简单等价成 `set_phase(SELECTION)`

以下内容根据源码结构整理补充

这两个名字看起来都像“回到 Selection”，但含义不一样。  
如果你只看 phase 值，很容易误判当前代码到底做没做完整回退。

- `set_phase(device, QI_PHASE_SELECTION, ...)`：只是切换阶段并重建超时
- `goto_selection(device)`：先停 `qi_ping`、关功率、关 ASK、清 `rx_packet / id_data / cfg_data / contract`，最后才切回 `Selection`

所以遇到异常、超时、EPT、协商失败这类“需要完整退回起点”的情况时，要看代码有没有走 `goto_selection()`，而不是只看 phase 值是不是变成了 `Selection`。

### 23.8 `rx_packet.valid` 为什么这里没看到互斥锁

以下内容根据源码结构整理补充

从当前实现看，`wlc_app_data_received_callback()` 会把收到的包写进 `ctx->rx_packet`，然后把 `valid` 置位；应用层阶段处理函数在消费完之后，再把 `valid` 清掉。

这套写法的意图是：

- 中断侧负责“投递一个完整包”
- 任务侧负责“读完后清标志”

这里不建议把它简单包装成“永远不需要并发保护”的结论。  
更稳妥的理解方式是：

- 当前工程就是按“单设备、单包缓冲、标志位交接”的方式设计的
- 只要沿用当前结构阅读和调试，就先按这个模型理解
- 如果以后要扩成多缓冲、队列化或多设备并行，再重新评估并发保护策略

## 24. 注意事项

这一章不讲新功能，也不讲新知识点，而是专门提醒你：  
哪些地方最容易因为“想当然”而把自己带偏。

### 24.1 不要把教程概念和源码实现机械等同

例如“Calibration 阶段”在文档概念上可以单独讲，  
但当前源码里它主要落在 `Qi_Power_Transfer` 内部逻辑里。

### 24.2 不要绕过 `wlc_lib`

如果以后你要补功能、接新硬件、改流程，这条提醒尤其重要。

如果以后你要补功能或改结构，尽量保持：

- 应用层调抽象接口
- 抽象接口再落到 BSP

否则项目分层会越来越乱。

### 24.3 不要只看函数名

这个提醒看起来简单，但真的很重要。  
因为嵌入式工程里，名字和真实行为不完全一致的情况并不少见。

当前工程里已经有一个很典型的例子：

- `app_10ms_task`
- 实际周期却是 2ms

所以排查时序问题时，一定以源码逻辑和实际参数为准。

### 24.4 不要把旧移植路径当成当前工程真实路径

以下内容根据源码结构整理补充

当前仓库里保留的某些旧 HTML 教程，记录的是历史移植过程，所以里面有些路径和当前工程已经不完全一致。  
这也是为什么这份统一教程一直强调“以当前仓库结构为准”。

例如当前实际可确认的是：

- 库文件在 `Lib/Tomato_wlc.h` 和 `Lib/Tomato_wlc.lib`
- 服务层源码不是平铺在 `User/WLC_server/src/`，而是分在：
  - `User/WLC_server/wlc_timer/src/`
  - `User/WLC_server/wlc_communications_physical/src/`
  - `User/WLC_server/wlc_power_bridge/src/`

所以以后如果你看到旧教程里的历史路径、历史文件名、历史“手写胶水层”说法，先不要直接照搬，一律先回到当前仓库目录确认。
